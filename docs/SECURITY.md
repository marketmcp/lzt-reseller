# Безопасность

## Чеклист перед публикацией

- [ ] Создан `.env` (он в `.gitignore` — не попадёт в git).
- [ ] `OWNER_PASSWORD` сменён со значения по умолчанию (`owner`). Лучше — `OWNER_PASSWORD_HASH`.
- [ ] Задан случайный `SESSION_SECRET` (`node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`).
- [ ] `LZT_TOKEN` только в `.env`, не во фронтенде.
- [ ] За HTTPS выставлен `SECURE_COOKIE=1`.
- [ ] В `web/config.js` нет реальных секретов (поле `admin.password` лучше держать в `.env`).

## Как устроена защита

**Токен LZT никогда не в браузере.** Фронтенд ходит в `/api/lzt/*`, бэкенд подставляет
`Authorization: Bearer <LZT_TOKEN>` из окружения. Клиент токен не видит.

**Авторизация владельца — серверная.** `POST /api/auth/login` сверяет пароль
(timing-safe, с анти-брутфорс задержкой) и ставит httpOnly-cookie с HMAC-подписью
(`SESSION_SECRET`). Подделать сессию нельзя. Без бэкенда (статичный демо-режим) вход
проверяется на клиенте — это **только для демо**, не для прода.

**Чувствительные операции — только владелец.** Прокси пропускает чтение листингов
публично (кэш), но `fast-buy`, `mafile`, `letters` требуют сессию владельца.

**Прочее.** Rate-limit на `/api` (60 req/мин на IP), защита от path traversal,
лимит тела запроса, заголовки `X-Content-Type-Options` / `X-Frame-Options` /
`Referrer-Policy`.

## Модель доверия (что безопасно, а что — демо)

- **Токены и креды никогда не в браузере.** Токен(ы) LZT, бот Telegram, CryptoBot — только
  в `.env`. Запросы к LZT идут через серверный прокси. Выдача аккаунта (`mafile`/`letters`)
  — только владельцу.
- **Общий каталог/федерация** содержит только публичные листинги (см. [CATALOG.md](CATALOG.md)).
  Никаких токенов/кредов. Покупка всегда валидируется вживую, проданное убирается из кэша.
- **Баланс пользователя в демо — на клиенте** (`USER.balance`). Это витрина-шаблон без
  реальных денег. Для боевого приёма платежей нужен серверный кошелёк: баланс, заказы и
  списания в БД, зачисление только по вебхуку/подтверждению платёжной системы (для CryptoBot
  оплата подтверждается сервером через `/api/pay/status`). Не полагайтесь на клиентский баланс
  как на источник истины в проде.
- **Уведомления о событиях** в проде стоит слать на стороне сервера из подтверждённого платежа
  (вебхук мерчанта), а не доверять клиентскому вызову. Эндпоинт `/api/notify/event`
  ограничен rate-limit.

## Если секрет утёк
Немедленно отзовите его у источника (LZT — перевыпустить токен; GitHub — Revoke токена;
бота — `/revoke` у @BotFather) и замените значение в `.env`. Секрет, попавший в git-историю
или в переписку, считается скомпрометированным.
