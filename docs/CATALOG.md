# Каталог: нода, пул токенов, федерация

Лимит LZT на поиск категорий — 20 запросов в минуту на токен. Чтобы быстро держать
актуальный каталог из тысяч аккаунтов, есть три механизма.

## 1. Пул токенов (быстрее в N раз)

Несколько токенов через запятую — запросы идут round-robin, каждый со своим лимитом:

```ini
LZT_TOKENS=token1,token2,token3
```

N токенов ≈ N×20 поисков/мин. Тайминги на токен настраиваются (`LZT_SEARCH_GAP_MS`).

## 2. Прогрессивная параллельная синхронизация

Фоновый обход идёт «вширь»: страница 1 ВСЕХ категорий, потом страница 2 всех, и т.д.
Поэтому категории наполняются одновременно — ни один «Steam на 100к» не блокирует
остальные. Период задаётся `CATALOG_SYNC_MIN` (мин). Владелец может запустить вручную:
`POST /api/catalog/sync`.

Результат лежит в общем кэше и отдаётся фронтенду через `GET /api/catalog`
(прогрессивно: витрина подтягивает товары по мере синка).

## 3. Федерация (общий кэш между нодами)

Кто уже спарсил каталог — делится им с другими, снимая нагрузку с API:

```ini
CATALOG_PEERS=https://friend-shop.example.com,https://another.example.com
```

Ваша нода периодически тянет публичный каталог у указанных пиров и сливает со своим.

### Безопасность федерации (важно)

- Делятся **только публичные листинги**: `item_id`, заголовок, цена, публичные атрибуты —
  ровно то, что и так видно любому на LZT. **Никогда** не передаются и не хранятся в этом
  кэше: токены, логины/пароли, maFile, письма, баланс, сессии. (Строгий whitelist полей —
  `server/catalog.js → publicItem`.)
- **Только pull, никакого push.** Вы сами выбираете, у кого тянуть. Чужая нода не может
  «затолкать» вам данные или отравить кэш — нет эндпоинта для приёма.
- **Покупка всегда валидируется вживую** через `fast-buy`. Если товар из кэша уже продан —
  сделка не пройдёт, а item автоматически убирается из каталога. Поэтому устаревший кэш
  не приводит к продаже «воздуха».
- `GET /api/catalog` отдаёт только публичные данные. Токены в браузер/пиры не уходят
  никогда — все запросы к LZT идут через серверный прокси.

## Перевалидация проданных

Когда покупка упирается в ответ LZT «sold/deleted», нода убирает товар из общего кэша.
Следующий обход/синхронизация добавит актуальные позиции.

## Эндпоинты

| Метод | Путь | Кто | Назначение |
|---|---|---|---|
| GET | `/api/catalog?cat=&q=&limit=&offset=` | публично | публичные листинги из кэша |
| GET | `/api/catalog/stats` | публично | размер кэша, прогресс, идёт ли синк |
| POST | `/api/catalog/sync` | владелец | запустить обход + подтяжку пиров |
